  - Intended: projection onto an axis from generic to deeply personal.
  - Heuristic values in [0.1–0.7]:
    - Generic activity videos (no "내", no clear personal history) → ~0.1.
    - Personal logs (notes, schedules, travel photos) → ~0.5–0.6.
    - Strongly personal with "내" or social + history ("내 여권 사진", "소연이가 추천했던 …") → ~0.7.

For later batches, you should:
- Keep the same set of cluster IDs where applicable.
- Keep numeric axis ranges roughly consistent (0–1), using higher values for more complex or more personalized queries.

### Modeling_Notes

- Likely high-impact features for MAL:
  - **Task & goal**: `QL_task_type`, `QL_goal_type`, `QL_recommender_task`, `QL_summary_required`, `QL_output_size_expectation`, `QL_requires_structured_output`.
  - **Time & urgency**: `QL_has_time_ref`, `QL_time_granularity`, `QL_has_future_ref`, `QL_urgency_level`, `QL_context_phrase_now_here`.

  - **Modality & domain**: `QL_data_modalities`, `QL_financial_domain`, `QL_health_activity_domain`, `QL_media_domain`, `QL_shopping_domain`, `QL_travel_location_domain`, `QL_calendar_schedule_domain`.
  - **Complexity proxies**: `QL_char_len`, `QL_word_len`, `QL_language_complexity_proxy`, `QL_emb_complexity_axis`, plus flags like `QL_requires_cross_modal`, `QL_temporal_comparison`.

- Recommended model families:
  - Start with an **interpretable GLM / linear regression** (or ordinal regression if MAL is ordinal buckets) using the core features plus a few key interactions (e.g., `QL_task_type × QL_urgency_level`, `QL_output_size_expectation × QL_data_modalities`).
  - Use **regularized linear models** (Lasso/Elastic Net) to handle correlated features and perform feature selection.
  - For capturing mild nonlinearity while staying interpretable, consider **shallow decision trees** or **Explainable Boosting Machines (EBMs)**.
  - If you later allow more complexity, **Random Forest or XGBoost** on these features can capture interactions, but keep interpretability aids (feature importance, SHAP).

- Evaluation plan:
  - Use **5-fold cross-validation** on the feature-augmented dataset.
  - If MAL is continuous: report **MAE**, **RMSE**, and **R²**.
  - If MAL is bucketed (e.g., ≤1s, 1–3s, 3–10s, >10s): report **macro F1**, **accuracy**, and a **within-1-bucket accuracy**.
  - Keep a **hold-out test set** (e.g., 20–30%) stratified by task type and domain to ensure generalization across query families.

### Interpretation_Summary

- These features capture key human factors that shape acceptable latency:
  - **Cognitive load & complexity**: Length, modality, history span, and complexity proxies reflect how much “work” the user expects the system to do. Heavier tasks (long-horizon analysis, structured outputs) generally allow higher MAL.
  - **Urgency & temporal coupling**: Time references and immediacy cues (오늘/주말/지금) model how “in the moment” the task is; tightly time-coupled tasks usually demand lower latency.
  - **Personalization & trust**: Depth of personalization and reliance on long personal history imply more sophisticated processing. Users often tolerate slightly more delay if they believe the system is doing rich, personalized reasoning.
  - **Domain & stakes**: Financial, calendar, health, and shopping domains differ in perceived importance and risk; MAL curves likely differ from entertainment or simple recall.

By learning how MAL varies across these interpretable dimensions, your model can generalize to new queries and support design decisions (e.g., when to prefetch, when to show progress indicators, or when to degrade gracefully) in a human-centered way.
